04 / 39
Arquitectura

CockroachDB: ranges

Antes de explicar por qué es más lento, necesitas saber cómo se ve el sistema.

  • Datos divididos en ranges (512 MB cada uno por defecto)
  • Cada range = un grupo Raft independiente, replicado en 3 nodos
  • Un nodo tiene el leaseholder por range — lee y coordina escrituras para ese range
  • Conforme crecen los datos, los ranges se dividen y rebalancean automáticamente
Nodo 1 Nodo 2 Nodo 3 ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Range A* │ │ Range A │ │ Range A │ │ Range B │ │ Range B* │ │ Range B │ │ Range C │ │ Range C │ │ Range C* │ └──────────┘ └──────────┘ └──────────┘ * = leaseholder (leases distribuidos entre nodos)
Cada escritura debe llegar al leaseholder de ese range, luego replicarse a un quórum de seguidores. Por eso “leaseholder” aparece en todas las próximas 20 diapositivas.

CAP Consistencia, Disponibilidad, Tolerancia a particiones. Durante una partición debes elegir C o A.
Teoría

Teorema CAP, bien explicado

  • En operación normal, puedes tener ambas C y A
  • Durante una partición de red, debes elegir una
  • Las particiones van a ocurrir: cables, switches, pausas de GC
  • La verdadera pregunta: ¿rechazar peticiones (CP) o servir datos obsoletos (AP)?
CP — CockroachDB
  • El lado minoritario deja de aceptar escrituras
  • Sin lecturas obsoletas, sin split-brain
AP — Cassandra
  • Ambos lados aceptan escrituras
  • Reconcilian después (last-write-wins)

CAP · en español sencillo

Imagina un Google Doc compartido

  • Consistencia: todos ven exactamente la misma versión, instantáneamente
  • Disponibilidad: siempre puedes editarlo, sin importar qué
  • Tolerancia a particiones: aunque internet separe a los usuarios en grupos, el doc sigue funcionando
Si la red se divide… CP

Bloquea la edición hasta restaurar la sincronización. Nadie ve datos obsoletos.

Si la red se divide… AP

Permite ediciones en ambos lados, pero arriesga conflictos que habrá que reconciliar después.

10 / 39
La respuesta OLTP

Para saldos bancarios, AP con last-write-wins
puede literalmente perder dinero.

Dos nodos aceptando transferencias independientemente en la misma cuenta = estado inconsistente. CP rechaza la escritura en vez de arriesgarse.

Crítica de Kleppmann: CAP trata de una consistencia muy específica (linearizabilidad) y una disponibilidad muy específica (todo nodo no-fallido responde). Los sistemas reales operan entre estos extremos.

Demo en vivo · demo_quorum.py

¿Cuándo CP realmente rechaza?

shell
$ uv run python demos/demo_quorum.py # Fase 1: 3/3 nodos → escrituras fluyen # Fase 2: kill 1 → quórum se mantiene, escrituras continúan # Fase 3: kill 2 → QUÓRUM PERDIDO → ¡errores! # Fase 4: restart 1 → quórum restaurado → escrituras reanudan # Fase 5: restart 2 → clúster completo

Quórum = ⌊N/2⌋+1. Para 3 nodos, eso es 2. Perder quórum → escrituras rechazadas, no corrompidas.

Observación

La garantía CP

Matar 1 de 3
  • Quórum mantenido (2/3)
  • Caída de TPS, luego recuperación
  • Cero errores — peticiones detenidas, no rechazadas
Matar 2 de 3
  • Quórum perdido (1/3)
  • 0 TPS — escrituras rechazadas
  • Errores cada ~3s (statement timeout)
Sorpresa: cuando matamos el nodo 3, TPS subió de ~1,300 a ~2,500. Si el nodo eliminado tenía un leaseholder en un range caliente, eliminarlo fuerza una re-elección — y el nuevo leaseholder puede estar menos cargado. La distribución puede accidentalmente corregir un cuello de botella.
✓ Restaurar solo 1 nodo → quórum restaurado → escrituras reanudan (~5s de ventana de recuperación). El clúster nunca arriesgó inconsistencia. Esto es lo que compras con la penalización de latencia.

Mejor caso

Gateway = Leaseholder

Cuando tu conexión cae en el leaseholder, el gateway se salta el salto de ruteo. Menos round trips = menor latencia.

16 / 39
Salto extra

Gateway ≠ Leaseholder

Gateway ≠ Leaseholder. Tu conexión llega al nodo 1, pero el leaseholder del range es el nodo 2. Eso es un round trip de red extra antes de que Raft siquiera comience.

17 / 39
Quórum El número mínimo de nodos que deben confirmar una escritura para considerarla committed. Para N nodos: ⌊N/2⌋+1.
Consenso

¿Cómo se ponen de acuerdo los nodos?

  • Elección de líder: números de término, solicitudes de voto, prevención de split-brain
  • Replicación del log: el líder agrega entradas, envía AppendEntries RPCs, espera ACK de mayoría
  • Seguridad: el nuevo líder siempre tiene todas las entradas committed
  • 3 nodos → quórum = 2. 5 nodos → quórum = 3. La latencia escala lentamente.
Cada escritura = líder + 1 round-trip de red a un seguidor (mínimo)
18 / 39
Raft · el camino feliz

Líder + 1 seguidor rápido = committed. El seguidor lento se pone al día después.

19 / 39
Consenso · fallo

¿Qué pasa si el líder muere?

  • Detección: los seguidores agotan el timeout de heartbeats, inician elección
  • Elección: el seguidor con el log más completo gana
  • Recuperación: el nuevo líder replica entradas no committed a nodos atrasados
  • ¿El viejo líder regresa? Trunca su log para coincidir con el nuevo líder
✓ Ningún dato committed se pierde. La propiedad de seguridad lo garantiza. Lo verificarás tú mismo en la demo de kill-node.
⚠ Pero: la elección toma segundos. Durante esa ventana, las escrituras a los ranges afectados se detienen. Este es el pico de latencia que observarás.

Escrituras single-key → Raft

¿Qué hay de transacciones que abarcan múltiples nodos?

Transferir $100 de la cuenta A (nodo 1) a la cuenta B (nodo 2).
Ambas deben tener éxito o ninguna. Entra: Two-Phase Commit.

21 / 39
El problema

Una transacción, dos ranges

Tu tabla accounts está dividida en la clave 501:

Range A (claves 1–500)
  • Leaseholder en Nodo 2
  • Tiene su propio grupo Raft
Range B (claves 501–1000)
  • Leaseholder en Nodo 3
  • Tiene su propio grupo Raft
una sola transferencia
BEGIN; UPDATE accounts SET balance = balance - 100 WHERE id = 42; -- Range A UPDATE accounts SET balance = balance + 100 WHERE id = 750; -- Range B COMMIT;
Dos nodos diferentes “poseen” estos datos. No puedes simplemente escribir en ambos independientemente — ¿qué pasa si uno tiene éxito y el otro falla? Crearías o destruirías dinero.
21 / 39
Coordinador El nodo que orquesta 2PC. En CockroachDB, siempre es el gateway — el nodo donde aterriza tu conexión SQL.
Tres roles

¿Quién coordina?

  • Gateway (coordinador) — tu conexión aterriza aquí (Nodo 1). Parsea SQL, planifica la query, descubre qué ranges están involucrados, orquesta 2PC.
  • Leaseholder A — posee Range A. Ejecuta lectura/escritura para claves 1–500. Ejecuta su propio grupo Raft.
  • Leaseholder B — posee Range B. Ejecuta para claves 501–1000. Grupo Raft separado.
✓ Cuando ambas claves están en el mismo range → el gateway envía todo a un solo leaseholder. Escritura Raft normal, sin 2PC. Por eso las transacciones “locales” son más rápidas.
¿Cross-range? El gateway debe hablar con 2 leaseholders y ejecutar el protocolo prepare/commit. Esos son los round trips extra que estás pagando.
22 / 39
2PC Two-Phase Commit — PREPARE a todos los participantes, luego COMMIT o ABORT atómicamente.
El protocolo

Two-Phase Commit

Dos round-trips de red en la ruta crítica. Cada uno es ~500 μs en el mismo datacenter.

23 / 39
Bloqueante Un protocolo es bloqueante si la falla de un solo nodo puede dejar a otros sin poder proceder — deben esperar indefinidamente la recuperación.
El problema

2PC es bloqueante

  • El coordinador envía PREPARE, los participantes responden YES y mantienen locks
  • El coordinador muere antes de enviar COMMIT
  • Los participantes quedan atascados: prometieron commit, no conocen la decisión
  • Locks mantenidos indefinidamente. Conocido desde los 1980s.
✓ CockroachDB resuelve esto con parallel commits: el coordinador escribe el estado de la transacción en un registro distribuido en paralelo con el último lote de escrituras, para que los participantes puedan descubrir el resultado independientemente.
El SQL distribuido moderno no ha eliminado 2PC. Ha hecho la ventana de fallo infinitesimalmente pequeña y recuperable.

SSI Serializable Snapshot Isolation — el default de CockroachDB. Detecta conflictos, aborta un lado.
Reintentos

Aislamiento serializable = reintentos en la aplicación

  • CockroachDB usa serializable por defecto (no read-committed)
  • Las transacciones en conflicto son abortadas, no encoladas
  • La aplicación debe capturar errores 40001 y reintentar toda la transacción
  • Postgres encola detrás de locks. CockroachDB dice “intenta de nuevo.”
patrón de reintento
for attempt in range(5): try: async with conn.transaction(): await do_transfer(conn, a, b) break except SerializationError: continue # retry
27 / 39
Insight · el problema

Por qué local alta contención colapsa

Local, alta contención: 9 TPS, 53% reintentos. He aquí por qué.

  • Claves {1–5}: emisor y receptor sacan del mismo pool
  • La cuenta 3 puede ser debitada en tx A y acreditada en tx B simultáneamente — conflicto cross-role
  • SSI detecta conflicto al commit → aborta al perdedor completamente (no lo encola)
  • Reintentos colisionan con otros reintentos → cascada exponencial
  • Un leaseholder serializa todo — no hay paralelismo posible
⚠ 53% de tasa de reintento significa que más de la mitad del trabajo de tu base de datos se desperdicia. El sistema está ocupado reintentando, no haciendo commit. Esto es colapso de contención SSI.
28 / 39
Insight · la solución

Por qué cross-range es 20× más rápido bajo contención

  • Débitos: {498–500} · Créditos: {501–503} — conjuntos de claves disjuntos
  • Un emisor nunca puede ser receptor — los conflictos cross-role son imposibles por diseño
  • Solo quedan conflictos same-role: dos txns ambas debitando la clave 498
  • CockroachDB resuelve conflictos same-role con un timestamp push, no un abort — sin cascada de reintentos
  • Cada leaseholder procesa su lado independientemente y en paralelo
✓ El range split en la clave 501 accidentalmente separó los roles de emisor y receptor en diferentes leaseholders. Este es el beneficio de la distribución: dominios de contención independientes. Solo funciona cuando tus claves calientes mapean limpiamente a diferentes ranges — una decisión de diseño de schema.
29 / 39
HLC Hybrid Logical Clock — combina tiempo físico con contadores lógicos para orden causal.
Relojes

Relojes y orden

En un solo Postgres: un contador XID incremental. En un sistema distribuido: ¿quién dice que la txn 1000 en nodo A ocurrió antes que la txn 1001 en nodo B?

  • Relojes Lamport: solo lógicos, sin relación con tiempo real. Suficientes para orden causal.
  • HLC (CockroachDB): tiempo físico + contador lógico. Tiempo real best-effort.
  • TrueTime (Spanner): GPS + relojes atómicos. Acota incertidumbre a microsegundos.
⚠ CockroachDB --max-offset = 500ms. Si los relojes se desfasan más de eso, el nodo se auto-termina. Una mala configuración de NTP ha tumbado clústeres en producción.

La respuesta

¿Cuándo distribuir?

  • Tolerancia a fallos entre dominios de fallo y disposición a pagar la penalización de latencia
  • El volumen de datos excede lo que un solo nodo puede almacenar
  • Escrituras multi-región son un requisito estricto
⚠ Si solo necesitas más TPS: réplicas de lectura, connection pooling o sharding a nivel de aplicación primero. Un solo Postgres en buen hardware frecuentemente supera a un CockroachDB de 3 nodos en throughput bruto.

Anexo

Serializable vs. Read Committed

CockroachDB (Serializable)
  • Conflictos detectados al momento del commit
  • La transacción perdedora es abortada
  • La app debe reintentar (código de error 40001)
  • Garantía más fuerte: sin anomalías
Postgres por defecto (Read Committed)
  • Conflictos resueltos por cola de locks
  • Transacciones esperan, no abortan
  • No se necesita lógica de reintento en la app
  • Más débil: phantom reads posibles
⚠ Diferentes niveles de aislamiento = diferente código de aplicación. Migrar de Postgres a CockroachDB no es solo un cambio de DSN si tienes cualquier contención.
