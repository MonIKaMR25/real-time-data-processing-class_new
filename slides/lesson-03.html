Distribution: the trade-off

  • CockroachDB: 3–8× slower per operation than single Postgres
  • You buy fault tolerance (quorum writes survive node death)
  • Every write pays consensus tax: leader + 1 network round-trip minimum
  • Contention + distribution = catastrophic (53% retry rate)
But we only showed one pattern: replicate everything via consensus. What if you want more TPS, not just more availability?

Shard A horizontal partition of data. Each shard holds a subset of rows, determined by a shard key.
Completing the OLTP picture

Sharding: partition for throughput

  • Replication (Raft, L2) = copies of the same data → fault tolerance, read scale
  • Sharding = split data by key → each shard handles independent traffic → write scale
  • CockroachDB’s “ranges” ARE shards (auto-split at 512 MB)
  • The cross-range experiment in L2 was sharding in disguise: disjoint key sets = independent contention domains
Shard A (users 1-1M) Shard B (users 1M-2M) Shard C (users 2M-3M) ┌──────────────────┐ ┌──────────────────┐ ┌──────────────────┐ │ Node 1 (leader) │ │ Node 3 (leader) │ │ Node 5 (leader) │ │ Node 2 (replica)│ │ Node 4 (replica)│ │ Node 6 (replica)│ └──────────────────┘ └──────────────────┘ └──────────────────┘ independent writes independent writes independent writes

How to shard Postgres

Database-level
  • Citus — Postgres extension, distributed tables
  • Vitess — MySQL sharding proxy (YouTube scale)
  • CockroachDB — auto-sharding via ranges
Application-level
  • Route by user_id % N in app code
  • Full control, but: cross-shard joins are your problem
  • Schema changes must hit every shard
⚠ Sharding solves write throughput. But now a business analyst asks: “What was total revenue last quarter?” — that query must fan out to every shard, scan every row, aggregate. There’s a better tool for that question.

The architecture question
Same CPU. Same RAM. Same disk.
How can anything be 67× faster?

It’s not magic. It’s a different physical layout for a different access pattern.

Heap tuple A complete row stored on an 8 KB page. All columns packed together, sequentially.
Physical layout

Row store: Postgres

Data is laid out as heap tuples. Each 8 KB page contains complete rows — all 17 columns packed together.

✓ Need 3 of 17 columns? Read only those 3. The other 14 are never loaded from disk. That’s the I/O reduction right there.
09 / 38
Columnar advantage #2

Compression: same type = compressible

Contiguous values of the same type compress dramatically. Typical ratios: 5–10×.

Technique
  • Dictionary encoding — payment_type has 4 values → 2-bit code per row
  • Delta encoding — sequential timestamps → store differences only
  • Run-length (RLE) — “1 repeated 847 times, 2 repeated 31 times”
Why rows can’t do this
  • A row mixes types: int, timestamp, varchar, numeric…
  • Adjacent rows have no type locality
  • Generic compression (LZ4, zstd) helps ~2×, not 10×
The 3 columns we need might be 2 GB uncompressed → ~300 MB on disk after columnar compression. Postgres reads ~20 GB for the same answer.

11 / 38
SIMD Single Instruction, Multiple Data — CPU processes 4–8 values per cycle (AVX2).
Columnar advantage #3

Vectorized execution

Postgres: tuple-at-a-time
  • Each operator processes one row
  • Passes it up to the next operator
  • 100M rows = 100M function calls
  • Branch prediction thrashes
  • CPU caches useless
DuckDB: vectorized (batches of 2048)
  • Operators process 2048 values at once
  • Tight loop, no per-row function calls
  • Data is contiguous → CPU prefetcher works
  • Compiler auto-vectorizes → SIMD
  • 5–10× CPU efficiency gain
Total speedup ≈ I/O reduction (10–60×) × CPU efficiency (5–10×) = 50–100×
12 / 38
Zone map Min/max metadata per column per row group. Checked before reading any data. Also called “data skipping” or “segment elimination.”
Columnar advantage #4

Zone maps: skip before reading

Each row group stores lightweight metadata: min, max, null count per column.

Query: WHERE pickup_datetime >= '2023-06-01' AND < '2023-09-01' Row Group 1: pickup_datetime min=2023-01-02, max=2023-03-28 → SKIP ✗ Row Group 2: pickup_datetime min=2023-03-28, max=2023-06-15 → SCAN (partial) Row Group 3: pickup_datetime min=2023-06-15, max=2023-09-02 → SCAN ✓ Row Group 4: pickup_datetime min=2023-09-02, max=2023-12-31 → SKIP ✗ Result: read 2 of 4 row groups → 50% I/O eliminated before touching data
✓ Zone maps cost nothing at write time — just a few bytes of metadata. Compare to Postgres B-tree indexes, which must be maintained on every INSERT. Column stores optimize reads at zero write cost.
13 / 38
The trade-off

Why column stores are terrible at OLTP

  • Insert one row = write to N separate column segments (one per column)
  • Point lookup = no B-tree index; must scan zone maps, then decompress segments
  • Update one value = decompress a compressed block, modify, recompress
  • Write model = batch-append; rows accumulate in a buffer, then flush to columnar segments periodically
OLTP optimizes for random access to complete rows
OLAP optimizes for sequential access to individual columns
No single layout serves both. This tension drives the entire architecture of modern data systems — and it’s why CDC (Lesson 5) and Kafka (Lesson 6) exist as bridges between the two worlds.

Query 2 · filtered aggregation

Zone maps in action

SQL
SELECT DATE_TRUNC('month', pickup_datetime) AS month, payment_type, COUNT(*) AS trips, AVG(fare_amount) AS avg_fare, SUM(tip_amount) AS total_tips FROM trips WHERE pickup_datetime >= '2023-06-01' AND pickup_datetime < '2023-09-01' GROUP BY month, payment_type ORDER BY month, payment_type;
✓ Data is roughly time-ordered in the Parquet files. DuckDB’s zone maps skip row groups outside the date range. Postgres does a full sequential scan — no zone maps exist in a heap.

Query 3 · high-cardinality GROUP BY

Hash table pressure

SQL
SELECT pickup_location_id, dropoff_location_id, COUNT(*) AS trips, AVG(fare_amount) AS avg_fare FROM trips GROUP BY pickup_location_id, dropoff_location_id ORDER BY trips DESC LIMIT 20;

Query 4 · window function

CPU-bound: rolling average

SQL
SELECT pickup_location_id, pickup_datetime, fare_amount, AVG(fare_amount) OVER ( PARTITION BY pickup_location_id ORDER BY pickup_datetime ROWS BETWEEN 100 PRECEDING AND CURRENT ROW ) AS rolling_avg FROM trips WHERE pickup_location_id = 132 -- JFK Airport ORDER BY pickup_datetime;

Experiment C · data ordering

Sort order makes zone maps click

SQL — DuckDB
-- Sorted by time (zone maps effective) CREATE TABLE trips_by_time AS SELECT * FROM trips ORDER BY pickup_datetime; -- Shuffled randomly (zone maps useless) CREATE TABLE trips_shuffled AS SELECT * FROM trips ORDER BY RANDOM();
  • trips_by_time: date range maps to contiguous row groups → most are skipped
  • trips_shuffled: every row group contains rows from every date → no skipping possible
  • Same query, same data, vastly different performance
✓ Sort order alignment between data layout and common query filters is why real data warehouses partition by date. It’s not just organizational — it enables segment elimination.

Awareness

Postgres can do columnar

The row/column distinction isn’t about the database brand. It’s about the storage engine.

  • Citus Columnar (formerly cstore_fdw) — columnar storage engine for Postgres
  • pg_analytics (ParadeDB) — brings DuckDB-style columnar analytics into Postgres
  • pg_duckdb — run DuckDB queries directly from Postgres
“Postgres is slow at analytics” is imprecise. Postgres’s row-oriented heap storage is slow at analytics. The query execution model is separate from the storage model, and both matter.

Synthesis

Why same hardware scans billions in OLAP
but chokes at 10k TPS in OLTP

Why OLAP is fast
  • I/O reduction — read only needed columns (5–10× less)
  • Compression — same-type values compress 3–10×
  • Segment elimination — zone maps skip irrelevant data
  • Vectorized execution — batch processing + SIMD
  • Late materialization — defer assembling rows until last moment
Why column stores can’t OLTP
  • Writes touch N segments per row
  • No B-tree indexes for point lookups
  • Compression blocks must decompress to modify
  • Batch-append model adds write latency
