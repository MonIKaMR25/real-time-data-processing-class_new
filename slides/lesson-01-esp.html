02 / 30
Autopsia

¿Qué acaba de pasar?

  • Abrimos 500 conexiones a un contenedor de Postgres
  • Cada conexión es un proceso del SO independiente (un “backend”)
  • Cada backend asignó ~50 MB de memoria privada
  • Total: ~25 GB demandados. Límite del contenedor: 4 GB.
  • El OOM killer de Linux eligió a Postgres. Todo perdido.
⚠ Sin apagado graceful. Sin flush del WAL (Write-Ahead Log). El trabajo en vuelo se pierde. Esto pasa en producción.
03 / 30
RSS Tamaño de conjunto residente — la memoria RAM física que un proceso ocupa realmente en este momento.
Investigación

¿Qué es una conexión?

  • Postmaster hace fork de un nuevo proceso por cada cliente
  • Cada backend: ~5–10 MB RSS (Tamaño de conjunto residente) en reposo, más bajo carga
  • work_mem, buffers temporales, buffers de ordenamiento — todo por backend
  • 500 conexiones = gigabytes solo en sobrecarga de procesos
  • El cambio de contexto mata el throughput más allá del punto óptimo
concurrencia_útil_máx ≈ (núcleos CPU × 2) + discos_efectivos
💡 Por eso existe pgbouncer. El connection pooling no es opcional a escala.

06 / 30
fsync Llamada al sistema que fuerza las escrituras en buffer a almacenamiento durable. Retorna solo cuando los bits están en disco.
IOPS I/O Operations Per Second — el límite de throughput de tu dispositivo de almacenamiento.
Cuello de botella 1

El muro del fsync

  • Cada COMMIT requiere un flush del WAL a almacenamiento durable
  • fsync() = “no retornes hasta que los bits estén en el plato”
  • NVMe SSD: ~100k IOPS → techo de ~100k commits síncronos/seg
  • Una conexión, un INSERT a la vez → un fsync por commit
⚠ Este es típicamente el primer cuello de botella que encontrarás. El disco se convierte en el punto de serialización.

TPS Transactions Per Second — la métrica principal de throughput para cargas OLTP.

Naive: una conexión, síncrono

load_naive.py
conn = psycopg.connect(DSN, autocommit=True) for _ in range(100_000): cur.execute("INSERT INTO orders ...") # cada INSERT → COMMIT → fsync → ACK (confirmación)

Esperado: ~1,800 TPS (p99 ≈ 1ms). El muro del fsync en acción.

08 / 30
Cuello de botella 1 · solución

Group commit: la concurrencia ayuda

  • Múltiples backends haciendo commit → Postgres agrupa sus flushes de WAL
  • Un fsync sirve a varios commits → costo amortizado
  • 10 conexiones: ~5× de mejora. 50: ~8×.
  • Rendimientos decrecientes a medida que crece la contención sobre el lock del WAL

11 / 30
Cuello de botella 1 · opción nuclear

Desactivando el freno del fsync

Lo que ganas
  • COMMIT retorna inmediatamente
  • El WAL se sigue escribiendo, solo no se hace fsync por commit
  • El background writer hace flush en lotes
Lo que pierdes
  • En crash: pierdes hasta 200ms de datos confirmados
  • A 50k TPS × 200ms = 10,000 txns perdidas
  • No es corrupción — solo commits confirmados-pero-perdidos
⚠ ¿No lo crees? demo_sync_loss.py inserta filas, crashea el contenedor, y cuenta lo que falta. Prueba real.

13 / 30
COPY Protocolo de carga masiva de PostgreSQL. Envía filas en binario, saltando el overhead por fila.
Cuello de botella 1 · eliminado

Batching: el protocolo COPY

  • En lugar de 1,000 viajes de ida y vuelta → una sola transferencia masiva
  • COPY transmite filas en binario, un flush de WAL por lote
  • Menos fsyncs, menos cambios de contexto, menos viajes de ida y vuelta de red
  • executemany: INSERTs en pipeline. Bien. COPY: stream binario. Excelente.
✓ El batching es gratis — sin compromiso de durabilidad. Solo menos operaciones por commit.

¿Qué se rompe ahora?

COPY paralelo escala — pero el WAL lock lo limita a ~3× con 8 workers. Ahora conocerás un cuello de botella donde la fracción serial es del 100%.

17 / 30
Teoría

Ley de Amdahl aplicada

Speedup = 1 / ( S + (1 − S) / N )

S = fracción serial. N = workers paralelos. En simple: cualquier trabajo que solo pueda ocurrir de uno-a-la-vez establece un techo duro en la aceleración total — agregar más conexiones no puede mover ese muro.

  • El fsync del WAL es serial — un disco, una cola de flush
  • Las esperas de lock son seriales — solo una txn tiene la fila
  • Incluso con conexiones infinitas, el speedup está acotado por 1/S
  • Si el 5% de tu carga es serial, speedup máximo = 20×
💡 Esto explica por qué las conexiones más allá del punto óptimo degradan el rendimiento. Más overhead, misma fracción serial.

Cuello de botella 2

El colapso de la fila caliente

  • Muchas transacciones hacen UPDATE a la misma fila (inventario, saldo)
  • Postgres usa una cola de espera — acceso serializado
  • Más allá de un umbral: más tiempo esperando que trabajando
  • Ninguna cantidad de hardware arregla la contención
SQL
-- 50 backends, todos pegando a la misma fila UPDATE orders SET amount = amount + 1 WHERE id = 1;
⚠ Las soluciones son arquitectónicas: locking optimista, sharding de la fila caliente, encolar actualizaciones. No puedes comprar la salida.

Cuello de botella 3

Shared buffers y el problema del doble buffering

  • Postgres administra su propio buffer pool (shared_buffers)
  • El SO también cachea las mismas páginas en el page cache
  • shared_buffers demasiado grande = cachear páginas dos veces
  • Regla general: 25% de la RAM — por esto
⚠ Para cargas con muchas escrituras: si el I/O de checkpoint se dispara, tu latencia de escritura se dispara. El checkpointer escribe páginas sucias desde shared_buffers al disco.

MVCC Multi-Version Concurrency Control — los lectores nunca bloquean a los escritores. Cada versión de fila tiene IDs de transacción xmin/xmax.
Cuello de botella 4

MVCC y tuplas muertas

  • UPDATE crea una tupla nueva. La vieja queda muerta.
  • Las tuplas muertas se acumulan → bloat de tabla → escaneos más lentos
  • Vacuum reclama espacio, pero compite con tu carga de trabajo
  • Los valores por defecto de autovacuum son conservadores — las escrituras pueden superarlo

Cerrando el círculo

¿Por qué murió?

  • Las conexiones son caras. Cada una es un proceso del SO con memoria privada.
  • La configuración importa. max_connections, work_mem, shared_buffers interactúan.
  • El cuello de botella rara vez es lo que crees. Mide, no adivines.
  • Postgres es suficientemente rápido — si entiendes las restricciones.
💡 De 1.8k a 227k TPS en el mismo hardware. Lo único que cambió fue cómo hablamos con la base de datos.

work_mem: derrame a disco

  • Cada operación de sort/hash obtiene hasta work_mem de RAM
  • Si excede el límite → derrama a archivos temporales en disco (lento)
  • Peligro: work_mem × conexiones activas × ops por query = memoria total
demo_work_mem.py
# 64kB work_mem → external merge sort (disk) SET work_mem = '64kB'; EXPLAIN ANALYZE SELECT * FROM orders ORDER BY amount DESC; -- Sort Method: external merge Disk: 4096kB # 256MB work_mem → in-memory quicksort SET work_mem = '256MB'; EXPLAIN ANALYZE SELECT * FROM orders ORDER BY amount DESC; -- Sort Method: quicksort Memory: 12800kB
⚠ Configurar work_mem = 256MB con 100 conexiones = hasta 25 GB solo para ordenamientos. La misma explosión de memoria que nos mató en la diapositiva 2.

Stack tecnológico — por qué cada cosa

PostgreSQL

El motor OLTP estándar de la industria. No estamos aprendiendo Postgres — lo usamos para aprender con qué lidia toda base de datos transaccional.

Redpanda

Compatible con Kafka, binario único, sin JVM. La misma API que usarás en producción — sin el overhead operativo que se comería nuestro tiempo de laboratorio.

PySpark

Aún el framework de procesamiento batch dominante en la industria. API en Python, ejecución distribuida real, funciona perfecto en un laptop en modo local.

PyFlink

Procesamiento de streams con estado exactly-once — la versión difícil del problema. PyFlink expone el modelo mental real sin esconderlo detrás de abstracciones.

ClickHouse

OLAP columnar que realmente corre en un laptop. Queries de mil millones de filas en segundos — hace imposible ignorar el tradeoff OLTP vs OLAP.

Docker + uv

Entornos reproducibles para depurar sistemas, no configuraciones. uv instala en segundos — nadie pierde 20 minutos con pip.
